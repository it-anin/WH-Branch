### BUG-PACK-001 — SKU+หน่วยซ้ำแถวที่สองสแกนต่อไม่ได้

- ระดับ: **P1**
- ช่วงงาน: พนักงานแพ็คกิ้ง
- สถานะ: ยืนยันจาก logic test ซ้ำ 2 รอบ
- เงื่อนไข: Picklist มี SKU และหน่วยเดียวกันมากกว่า 1 แถว และอยู่กับพนักงานคนเดียวกัน
- ขั้นตอนทำซ้ำ:
  1. สร้างสองแถว `DUP/ชิ้น` จำนวน 2 และ 3 โดยใช้บาร์โค้ดเดียวกัน
  2. สแกนแถวแรกให้ครบ 2
  3. สแกนบาร์โค้ดเดิมเพื่อเริ่มยอดอีก 3
- Expected: ระบบเลือกแถวซ้ำที่ยังไม่ครบและรับสแกนต่อ
- Actual: `catalog.find()` และ `items.find()` เลือกแถวแรกที่ครบแล้วเสมอ จากนั้นแจ้ง “ครบแล้ว” ทั้งที่ยังเหลืออีกแถว
- ผลกระทบ: งานแพ็คหยุดหรือแพ็คไม่ครบ Picklist
- จุดโค้ดที่น่าสงสัย: `src/screens/PackScanC.jsx:496-499`

### BUG-BR-001 — SKU ซ้ำสามารถยืนยันรับด้วยจำนวนต่ำกว่ายอดรวม

- ระดับ: **P1**
- ช่วงงาน: สาขารับสินค้า
- สถานะ: ยืนยันจาก logic test ซ้ำ 2 รอบ
- เงื่อนไข: ภายในลังมี SKU เดียวกัน 2 แถว เช่น 2 และ 3 หน่วยฐาน
- ขั้นตอนทำซ้ำ:
  1. สร้าง `boxItems` สองแถว SKU เดียวกัน โดย `gotBase` เท่ากับ 2 และ 3
  2. สแกน SKU นี้รวมเพียง 3 ครั้ง
  3. กดยืนยันตรวจรับ
- Expected: ต้องสแกนรวม 5 หน่วย
- Actual: `scanCounts` เก็บค่าต่อ SKU เพียงช่องเดียว และนำค่า 3 ไปเทียบกับแต่ละแถวแยกกัน จึงถือว่าทั้งแถว 2 และแถว 3 ครบ
- ผลกระทบ: สาขารับสินค้าไม่ครบแต่ระบบอนุมัติผ่าน
- จุดโค้ดที่น่าสงสัย: `src/screens/BranchReceive.jsx:721-725`, `src/screens/BranchReceive.jsx:754-760`

### BUG-OUT-001 — ปุ่ม +/- แก้สินค้าแล้วจำนวนหน่วยฐานไม่เปลี่ยน

- ระดับ: **P1**
- ช่วงงาน: คลังแก้ปัญหาสินค้า
- สถานะ: ยืนยันจาก logic test ซ้ำ 2 รอบ
- ขั้นตอนทำซ้ำ:
  1. เปิดลังที่สาขาแจ้งปัญหาและคลังกด `+` หรือ `−` ในหน้าปัญหาแบบรวดเร็ว
  2. กด “อนุมัติ” โดยไม่เข้าหน้าแก้ไขตารางเต็ม
  3. กลับไปดูจำนวนที่สาขา
- Expected: `qty` และ `gotBase` เปลี่ยนให้สอดคล้องกับหน่วยที่สแกนจริง
- Actual: `adjustQty()` เปลี่ยนเฉพาะ `qty`; `resolveProblem()` ไม่คำนวณ `gotBase` ใหม่ ขณะที่สาขาเลือกอ่าน `gotBase` ก่อน `qty`
- ผลกระทบ: คลังเห็นว่าแก้แล้ว แต่สาขายังได้รับจำนวนเดิม โดยเฉพาะสินค้าที่ factor มากกว่า 1
- จุดโค้ดที่น่าสงสัย: `src/screens/BoxClosedLabel.jsx:320-334`, `src/screens/BranchReceive.jsx:752-755`
- หมายเหตุ: เส้นทาง “แก้ไขรายการสินค้า” แบบเต็มมีการคำนวณ `gotBase` แล้ว แต่ปุ่ม +/- ทางลัดยังไม่มี

### BUG-SYNC-001 — การเขียนจากเครื่องเก่าสามารถลบ field ที่อีกเครื่องเพิ่งบันทึก

- ระดับ: **P1**
- ช่วงงาน: Sync หลายเครื่อง
- สถานะ: ยืนยันกับ Firestore Emulator ซ้ำ 3 รอบ
- ขั้นตอนทำซ้ำ:
  1. เครื่อง A และ B อ่านลังสถานะ `exported` พร้อมกัน

- หลักฐาน Emulator: เอกสาร `QA-RACE-1` หลัง write ที่สองมี `note` แต่ไม่มี `receivingBy`
- จุดโค้ดที่น่าสงสัย: `src/App.jsx:249-269`; `setItemsByBox()` มีรูปแบบ full-document write คล้ายกันที่ `src/App.jsx:273-280`

## สิ่งที่ทดสอบผ่าน

- `npm run build` ผ่านทั้งโปรเจกต์ต้นฉบับและสำเนา QA
- Factor จาก R05.106 และกฎ `Nหน่วย` เช่น `3ลัง` คำนวณถูก
- Picklist ปกติแยกโซนจาก Location และเบิกด่วนเข้าโซนพิเศษ
- การหา branch ของลังแบบสาขาเดียวและ fallback เมื่อปนหลายสาขาทำงานตามโค้ด
- `catalogSig` เปลี่ยนเมื่อเนื้อหาเปลี่ยนแม้จำนวนแถวเท่าเดิม
- การหักยอดที่แพ็คแล้วแบบสะสมสำหรับ Picklist แถวซ้ำทำงานถูกใน `buildPackItems()` และ `catalogPackStatus()`
- Barcode index รองรับหลายหน่วย และ `matchBarcode()` ไม่ crash เมื่อ field barcode หาย
- Firestore Auth/Firestore Emulator เปิดใช้งานและรับ read/write โดยไม่แตะโปรเจกต์จริง

## รายการที่ถูก BLOCKED

- Login และ navigation จริงบน Desktop/Android
- การสแกนผ่าน input, Enter, `wh-scan` Broadcast และการรักษา focus
- การดาวน์โหลด/ตรวจไฟล์ Text จริงจากปุ่ม UI
- การพิมพ์ฉลากและการตรวจ responsive layout
- Blind receiving, dialog ตรวจแทน และ problem flow แบบคลิกจริงหลายหน้าต่าง
- Offline/reconnect ผ่าน browser context
- การทดสอบฮาร์ดแวร์ scanner และปุ่มบน PDA จริง

สาเหตุ: Browser runtime ไม่มี browser backend (`agent.browsers.list()` คืนรายการว่าง) และไม่มี PDA เชื่อมต่อ จึงไม่ใช้ Playwright/เครื่องมืออื่นแทนเพื่อหลีกเลี่ยงการออกนอกวิธีทดสอบที่กำหนด

## Smoke checklist สำหรับ PDA จริง

1. เปิดแอป Android และล็อกอินด้วยรหัส WAREHOUSE/SSS ให้ถูกโหมด
2. ยืนยันว่าต้องกด “เปิดลัง” ก่อนสแกน และเวลาเปิดลังถูกบันทึก
3. ยิงบาร์โค้ดผ่านฮาร์ดแวร์ ตรวจเสียงสำเร็จ/ผิดและ focus หลัง dialog
4. สแกนสินค้าหลายหน่วย เลือก LOT และปิดลัง
5. ให้ PDA สาขาสองเครื่องสแกนลังเดียวกัน ตรวจ dialog “ตรวจแทน” และการเด้งเครื่องเดิมออก
6. ตัด Wi-Fi ระหว่างนับ เชื่อมกลับ แล้วตรวจว่าไม่มีการยืนยันผิดลังหรือสถานะย้อนกลับ
7. แจ้ง damaged พร้อมรูป ตรวจว่ารูปและ note ปรากฏที่คลัง
8. ทดสอบ incomplete recheck ด้วยพนักงานทั่วไปและเภสัช
9. อนุมัติรับ แล้วสแกนลังเดิมซ้ำ ต้องถูกบล็อก
10. ตรวจ Firestore ว่า `boxes`, `boxItems`, `progress` และ `config/receive` ไม่มีเอกสารกำพร้า
